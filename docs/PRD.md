# 无尽炼金 (Infinite Craft) — 产品需求文档

## 1. 项目概述

### 1.1 产品定位

一款基于 AI 驱动的元素合成 Web 应用。用户从"金、木、水、火、土"五种基础元素出发，通过拖拽将两种元素组合，由 AI 实时生成合成结果。合成产物自动进入用户元素库，可被无限复制、继续合成，最终构建出一棵庞大的合成树。

### 1.2 核心玩法循环

```
选择元素 → 拖拽合成 → AI 返回新元素 → 元素入库 → 继续合成
```

### 1.3 目标用户

- 喜欢沙盒/创意类游戏的玩家
- 对中国传统五行文化感兴趣的用户
- 休闲探索型用户

## 2. 功能需求

### 2.1 元素系统

#### 2.1.1 基础元素

| 元素 | Emoji | 类别   |
| ---- | ----- | ------ |
| 金   | 🪙    | 五行   |
| 木   | 🪵    | 五行   |
| 水   | 💧    | 五行   |
| 火   | 🔥    | 五行   |
| 土   | 🪨    | 五行   |

> 基础元素的 Emoji 为预设，后续合成元素的 Emoji 由 AI 分配。

#### 2.1.2 元素属性

每个元素包含以下字段：

| 字段         | 类型       | 说明                                       |
| ------------ | ---------- | ------------------------------------------ |
| `id`         | string     | 唯一标识（如 UUID 或自增 ID）              |
| `name`       | string     | 元素名称（如"蒸汽"）                       |
| `emoji`      | string     | AI 选定的 Emoji 表情                       |
| `categories` | string[]   | 元素类别列表（由 AI 分配）                 |
| `isBase`     | boolean    | 是否为基础元素                             |
| `discoveredAt` | timestamp | 首次发现时间                             |

#### 2.1.3 元素显示

元素以卡片形式展示，格式为：

```
┌──────────┐
│  🔥 火   │
└──────────┘
```

- Emoji 在左，名称在右
- 卡片可拖拽、可复制
- 新发现的元素有"✨ 新发现"标记动画

### 2.2 合成机制

#### 2.2.1 拖拽合成

- 用户从元素库中拖出一个元素到合成区域（工作台）
- 将另一个元素拖拽至已有元素上方，两个元素重叠时触发合成
- 支持相同元素自我合成（如 火 + 火 = ?）
- 合成过程中显示加载动画

#### 2.2.2 合成结果处理

| 场景               | 行为                                           |
| ------------------ | ---------------------------------------------- |
| 全新元素           | 播放"新发现"动画，元素加入元素库               |
| 已发现的元素       | 直接显示结果，不重复加入元素库                 |
| 无法合成           | AI 返回 `null`，提示"这两种元素无法合成"       |

#### 2.2.3 合成缓存

- 相同的两个元素组合（A+B 与 B+A 视为同一配方）应缓存结果
- 首次合成时请求 AI，后续相同合成直接返回缓存结果
- 缓存存储在后端数据库中，所有用户共享

### 2.3 元素库（侧边栏）

#### 2.3.1 基本功能

- 展示用户已发现的所有元素
- 支持搜索/筛选（按名称、类别）
- 支持按类别分组显示
- 支持按发现时间排序
- 显示已发现元素总数

#### 2.3.2 元素复制

- 元素库中的元素可以无限次拖出到工作台
- 拖出时为"复制"操作，原始元素仍留在元素库中
- 工作台上的元素可以被拖拽移动、删除

#### 2.3.3 类别系统

类别由 AI 在合成时动态分配，每个元素可拥有 1~3 个类别。

预设类别池（AI 可从中选择，也可创建新类别）：

```
五行、自然、天气、动物、植物、食物、饮品、
矿物、工具、武器、建筑、交通、科技、神话、
人物、情感、艺术、音乐、抽象概念、天体、
地理、化学、物理、生物、历史、文学……
```

> AI 可根据合成结果自由创建新类别，但应尽量复用已有类别。

### 2.4 合成记录

#### 2.4.1 元素图鉴

维护一个全局列表，记录所有已被发现的元素：

| 列名         | 说明                     |
| ------------ | ------------------------ |
| Emoji + 名称 | 元素展示                 |
| 类别         | 该元素所属类别           |
| 发现时间     | 首次被发现的时间         |
| 发现方式     | 配方（如：水 + 火 → 蒸汽）|

#### 2.4.2 合成树 / 配方表

记录所有合成配方：

```
配方结构：
{
  elementA: "水",
  elementB: "火",
  result: "蒸汽",
  discoveredBy: "user_xxx",   // 可选
  discoveredAt: "2026-03-25"
}
```

功能要求：
- 列表视图：以表格形式展示所有配方
- 树形视图：可视化展示元素的合成路径（从基础元素到高阶元素）
- 支持搜索：输入元素名称，查看该元素的所有合成来源和去向
- 支持反向查询：选择一个元素，查看它可以与哪些元素继续合成

### 2.5 工作台

#### 2.5.1 布局

- 主区域为自由拖拽的画布（工作台）
- 左侧/下方为元素库面板（可折叠）
- 顶部为工具栏

#### 2.5.2 工作台功能

- 无限画布，支持缩放和平移
- 元素可自由拖拽放置
- 双击工作台空白区域可清空
- 右键元素可删除/查看详情
- 工作台状态自动保存

## 3. AI 集成

### 3.1 Prompt 设计

向 AI 发送的请求格式：

```
你是一个元素合成游戏的裁判。用户将两种元素合成，你需要返回合成结果。

规则：
1. 优先返回符合现实逻辑、物理属性、功能关系或常识因果的结果，不要为了“创意”强行发散
2. 优先考虑直接混合产物、材料加工、物态变化、自然现象或现实场景中会出现的对象
3. 避免过于抽象、牵强、神话化、网络梗化或跨越过多推理步骤的结果
4. 如果两个元素之间缺乏直接且清晰的联系，返回 null，不要硬凑
5. 结果应该是一个具体名词或很短的名词短语（2-4个字为佳）
6. 如果存在多个候选结果，优先选择最常见、最朴素、最容易被玩家直觉接受的那个
7. 为结果选择一个最能代表它的 Emoji
8. 为结果分配 1-3 个类别
9. reason 用一句简短的话说明逻辑关系，不要写成花哨文案

元素A: {elementA}
元素B: {elementB}

请以 JSON 格式返回：
{
  "name": "合成结果名称",
  "emoji": "🎯",
  "categories": ["类别1", "类别2"],
  "reason": "简短解释为什么这样合成"
}

如果无法合成，返回：
{
  "name": null
}
```

### 3.2 AI 模型选择

支持以下模型接入方式（按优先级）：

1. **OpenAI API** (GPT-4o / GPT-4o-mini) — 首选
2. **其他兼容 OpenAI 格式的 API** — 便于接入国产模型
3. 预留模型切换配置，便于后续扩展

### 3.3 AI 响应处理

- 解析 AI 返回的 JSON
- 校验字段完整性（name、emoji、categories）
- Emoji 合法性验证
- 错误重试机制（最多 3 次）
- 超时处理（10 秒超时）

## 4. 数据模型

### 4.1 Element（元素）

```typescript
interface Element {
  id: string;
  name: string;
  emoji: string;
  categories: string[];
  isBase: boolean;
  discoveredAt: Date;
}
```

### 4.2 Recipe（配方）

```typescript
interface Recipe {
  id: string;
  elementA_id: string;   // 输入元素A
  elementB_id: string;   // 输入元素B
  result_id: string;     // 结果元素（null 表示无法合成）
  discoveredAt: Date;
}
```

### 4.3 UserDiscovery（用户发现记录）

```typescript
interface UserDiscovery {
  userId: string;
  elementId: string;
  discoveredAt: Date;
  recipeId: string;      // 通过哪个配方发现的
}
```

## 7. 交互细节

### 7.1 拖拽行为

| 操作                     | 行为                                    |
| ------------------------ | --------------------------------------- |
| 从元素库拖出元素         | 复制一份到工作台                        |
| 工作台内拖动元素         | 自由移动位置                            |
| 元素 A 拖到元素 B 上     | 触发合成请求                            |
| 合成中                   | 两个输入元素消失，显示合成动画          |
| 合成成功                 | 结果元素出现在原位置，新元素入库        |
| 合成失败 (null)          | 两个输入元素回到原位，显示"无法合成"提示 |
| 右键点击工作台上的元素   | 显示上下文菜单（删除/查看详情）         |
| 双击工作台空白处         | 清空工作台                              |

### 7.2 动画效果

- **合成中**：两个元素向中心靠拢 + 旋转 + 发光
- **新发现**：结果元素放大弹出 + ✨ 粒子效果 + "首次发现"横幅
- **已知元素**：结果元素淡入
- **合成失败**：元素抖动 + 回弹

### 7.3 快捷操作

- `Delete` / `Backspace`：删除选中的工作台元素
- `Ctrl + Z`：撤销上一步操作
- 鼠标滚轮：缩放画布
- 中键拖拽 / 空格+左键拖拽：平移画布

## 8. 非功能需求

### 8.1 性能

- AI 请求响应时间 < 5 秒（含网络延迟）
- 相同配方的缓存命中应 < 50ms
- 工作台应支持同时放置 50+ 元素而不卡顿

### 8.2 数据持久化

- 用户的已发现元素列表需要持久化
- 工作台状态自动保存（页面刷新后恢复）
- 全局配方库持久化

### 8.3 兼容性

- 支持现代浏览器（Chrome、Firefox、Edge、Safari）
- 桌面端优先，移动端后续适配
- 移动端可考虑点击选择替代拖拽

## 10. 待讨论 & 决策点

| # | 问题                                           | 备选方案                            | 建议     |
|---|------------------------------------------------|-------------------------------------|----------|
| 1 | 类别系统：预设固定类别 vs AI 自由创建？        | A: 纯预设 / B: 纯AI / C: 混合      | **C 混合**：提供预设池，AI 可创建新类别 |
| 2 | 数据共享：全局共享 vs 用户独立？               | A: 全局 / B: 独立                   | **A 全局**：配方全局共享，发现记录独立  |
| 3 | 架构选型：全栈 vs 纯前端？                     | A: 全栈 / B: 纯前端                 | 视需求定 |
| 4 | "无法合成"的概率：是否所有组合都应产生结果？   | A: 全部可合成 / B: 允许失败         | **A 全部可合成**：更有趣味性          |
| 5 | 合成结果一致性：同配方是否永远产出同一结果？   | A: 固定 / B: 随机                   | **A 固定**：缓存友好，体验一致        |
| 6 | 是否需要用户登录系统？                         | A: 需要 / B: 匿名（本地存储）      | 视需求定 |
